Feature Flags

Deploy code without releasing features. Ship faster, break less, roll back instantly.

What is a feature flag?

A feature flag (also called a feature toggle or feature switch) is a technique that lets you enable or disable functionality at runtime — without deploying new code. Instead of tying a release to a deployment, you wrap the feature in a conditional: 

import { FlagClient } from './flag-client';

const flags = new FlagClient({ endpoint: 'http://flags.internal' });
await flags.init();

if (flags.isEnabled('new-checkout-flow', { userId: user.id })) {
  renderNewCheckout();
} else {
  renderOldCheckout();
}
client := flagclient.New("http://flags.internal")
if err := client.Init(); err != nil {
    log.Fatal(err)
}

enabled, _ := client.IsEnabled("new-checkout-flow", map[string]string{
    "userId": user.ID,
})

if enabled {
    renderNewCheckout(w, r)
} else {
    renderOldCheckout(w, r)
}

The flag's value is controlled externally by an API or config store. Your app doesn't know or care why the flag is on or off. It just asks and reacts.

How flags are evaluated

When your app calls isEnabled(flagKey, context), the flag service runs the flag's rules against the context you passed. Rules are evaluated in order — the first match wins.

1
Override list

Explicit user IDs that always get the feature. Used for internal staff and beta testers.

2
Segment rules

Attribute-based targeting — e.g. "all users on the enterprise plan" or "all users in Germany."

3
Percentage rollout

Release to a consistent slice of users via a hash function. Same user, same result, every time.

Types of feature flags

Release flags

Permission flags

Percentage rollout and sticky bucketing

When you roll out to 10% of users, those users need a consistent experience — not randomly see the feature on one page load and miss it on the next. This is solved with a hash: 

import crypto from 'crypto';

function getUserBucket(flagKey: string, userId: string): number {
  const input = `${flagKey}:${userId}`;
  const hash = crypto.createHash('md5').update(input).digest('hex');
  return parseInt(hash.slice(0, 8), 16) % 100; // 0–99
}

// User is in rollout if their bucket < rolloutPercentage
const inRollout: boolean = getUserBucket('new-checkout', userId) < 10;
import (
    "crypto/md5"
    "encoding/binary"
    "fmt"
)

func getUserBucket(flagKey, userID string) int {
    input := fmt.Sprintf("%s:%s", flagKey, userID)
    sum := md5.Sum([]byte(input))
    // Take first 4 bytes as a uint32, mod 100
    num := binary.BigEndian.Uint32(sum[:4])
    return int(num % 100) // 0–99
}

// User is in rollout if their bucket < rolloutPercentage
inRollout := getUserBucket("new-checkout", userID) < 10

Because the hash is deterministic, user_42 always lands in the same bucket across every server and every request. Raise the rollout from 10% to 50% and users already in it stay in — they just get more company.

Why not just deploy the feature?

A deployment rollback takes 10–20 minutes and often requires waking someone up. A flag rollback is flipping a boolean — it propagates in seconds. Feature flags decouple deploying code from releasing features, which is one of the highest-leverage habits an engineering team can build.
